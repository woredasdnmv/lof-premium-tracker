## M1 体验打磨

> 目标：修复已知缺陷，移动端可用性显著改善，加载等待不再让用户焦虑。先建立导航页提升网站可扩展性。

### 1.0 导航页

**当前问题：** 用户直接进入 LOF 数据页，网站没有统一的入口层。未来扩展 ETF、货币基金、REITs 等模块时，缺乏承接页面。

**方案：** 新建导航页作为网站首页（`index.html`），LOF 数据页迁至 `/lof.html`。导航页秒开（无 loader、无数据请求），点击基金模块后进入数据页。

### 1.2 Toast 显示时长延长 ✅ 已完成

~~**当前问题：**~~ Toast 提示固定 1 秒后消失（`app.js` 中 `setTimeout` 设为 1000ms）。设置应用 toast 文字较长（如"设置已应用：溢价率≥5%，成交额≥100万，佣金万1.5 最低5元 最大投入1000元"），1 秒用户读不完。

**方案：** 根据 toast 文字长度动态计算显示时长。公式建议：`max(2000, 文字长度 × 50ms)`，最少 2 秒，最多 5 秒。

**涉及文件：** `js/app.js` — `showToast()` 方法

**验收标准：**
- [ ] 短 toast（如"复制成功"）显示 ≥ 2 秒
- [ ] 长 toast（如设置应用提示）显示足够阅读完毕
- [ ] toast 消失动画保持流畅

### 1.4 首屏加载体验全面优化

**当前问题：** 首次连接时用户看到一个静态 spinner + "正在连接服务器..." + "首次加载可能需要 10-30 秒"。重试期间 loader 文字会更新（"连接中... (1/5)"），但整体体验仍然粗糙——用户不确定系统是否卡死，等待焦虑。

**子任务：**

#### 1.4.1 骨架屏（Skeleton Screen）

**方案：** 在 loader 显示期间，同时在主内容区渲染一个简化的骨架屏——模拟表格结构（灰色占位块），让用户感知"页面结构已加载，数据正在填充"。

**验收标准：**
- [ ] 首次加载时，loader 下方可见表格骨架屏（灰色条块模拟行）
- [ ] 数据就绪后骨架屏平滑过渡到真实表格
- [ ] 骨架屏适配深色/浅色模式

**预计耗时：** 1h

#### 1.4.2 进度条 + 预计等待时间

**方案：** 替换静态 spinner 为带进度信息的加载组件：

- 连接阶段：显示"正在连接服务器..." + 不确定进度条（indeterminate）
- 重试阶段：显示"第 N/5 次重试，X 秒后重试..." + 倒计时数字
- 数据加载阶段：显示"正在获取基金数据..." + 确定进度条（如已知总数）
- 渲染阶段：显示"正在渲染..."

**验收标准：**
- [ ] 加载过程至少展示 3 个阶段的文字变化
- [ ] 用户能感知到系统在工作而非卡死
- [ ] 重试倒计时可见

**预计耗时：** 1.5h

#### 1.4.3 等待小游戏：财神接元宝

---

**设计原则：** 游戏绝不拖慢数据加载。游戏运行在主线程的空闲时段（requestAnimationFrame），数据请求不受任何影响。游戏区域限定在 200×200px，替换原 spinner 位置，不改变 loader 整体布局。

---

##### 1.4.3.1 状态机与时间线

```
页面加载开始
  │
  ├─ 0 ~ 3 秒 ──→ 标准 loader：spinner + 加载文字 + 进度条（保持专业第一印象）
  │
  ├─ 3 秒 ──→ spinner 区域自动替换为游戏 Canvas（200×200px）
  │            加载文字变为："数据加载中，玩个小游戏打发时间吧～"
  │            进度条保持可见，位于游戏区域下方
  │
  ├─ 游戏中 ──→ 3 个退出路径：
  │   ├─ 路径A：用户点右上角 "×" → 游戏销毁 → 恢复 spinner → 回到纯 loader
  │   ├─ 路径B：60 秒倒计时结束 → 游戏结束弹窗（得分 + 再来一局/退出游戏）
  │   └─ 路径C：数据加载完成 → 游戏立即停止 + 销毁 → 整个 loader 淡出（300ms）→ 主界面淡入
  │
  └─ 数据加载完成时的特殊处理：
      - 如果游戏正在运行：暂停游戏帧循环 → 显示半透明通知条（"数据已就绪，游戏结束后自动进入"）
        → 游戏继续但计时加速 3 倍 → 倒计时归零后自动跳转
      - 如果用户在游戏结束弹窗中：点「退出游戏」→ 立即跳转数据页
      - 如果用户已关闭游戏（路径A）：loader 淡出 → 直接进入数据页
```

##### 1.4.3.2 游戏 UI 布局（200×200px 游戏区）

```
┌──────────────────────────┐
│ ⏸ 暂停  倒计时 60s  🏆 0 │  ← DOM 层：顶部状态栏（30px）
│                          │
│    ⬇️ 掉落物              │
│    🪙 铜钱                │
│                          │  ← Canvas 层：游戏主体（140px）
│         🦢 黑天鹅         │
```

##### 1.4.3.7 技术架构

**新增文件：**

```
js/game.js                    # 游戏引擎（独立模块，不依赖 LofFundMonitor）
├── class CaishouGame {
│     constructor(canvas, config)  # 初始化 Canvas 2D 上下文、素材、状态
│     start()                      # 启动帧循环
│     pause() / resume()           # 暂停/继续
│     destroy()                    # 停止帧循环 + 清理事件监听 + 释放 Canvas
│     onScoreChange(callback)      # 得分变化回调（通知 DOM 层更新分数）
│     onCountdown(callback)        # 倒计时回调
│     onGameOver(callback)         # 游戏结束回调
│   }
└── 掉落物生成器、碰撞检测、粒子特效均为内部方法

css/game.css                  # 游戏 UI 样式（弹窗、按钮、进度条保留在 style.css 中）
```

**与主加载流程的通信接口（game.js 不 import 任何业务模块）：**

```js
// 主流程 → 游戏
game.start()          // 启动
game.pause()          // 暂停
game.destroy()        // 销毁（数据就绪时调用）

// 游戏 → 主流程（回调）
onScoreChange(score)  // 更新 DOM 分数显示
onCountdown(seconds)  // 更新 DOM 倒计时
onGameOver(score)     // 弹出游戏结束弹窗
onClose()             // 用户点 ×，销毁游戏回到 loader
```

**性能约束：**
- Canvas 尺寸固定 200×140px（逻辑像素），不支持缩放
- 帧率上限 60fps，移动端自动降到 30fps（检测 `navigator.hardwareConcurrency` 或屏幕宽度）
- 每个掉落物为预渲染的 Image 对象（SVG → Image），不做实时矢量绘制
- 飘分特效使用简单的 DOM 绝对定位（CSS transition），不占用 Canvas 重绘
- 游戏暂停或销毁时必须 `cancelAnimationFrame`，零后台消耗

## M2 性能与缓存

> 目标：浏览器缓存策略优化，重复访问不再全量下载。

### 2.1 静态资源缓存策略

**当前架构说明：**

```
CF Pages（静态资源 + _headers 文件）    Railway（API + CORS 已在 Functions 层处理）
├── index.html                         ├── /api/funds
├── css/style.css                      ├── /health
├── js/*.js                            └── /api/funds/<code>/chart
└── assets/*
```

CORS 已通过 `functions/api/[[path]].js` 在每个响应中设置了 `Access-Control-Allow-Origin: *`，无需额外处理。本任务只涉及 `Cache-Control` 头的添加。

**当前问题：** 静态资源（CSS/JS/图片）没有配置缓存头，用户每次访问都重新下载全部文件。

**方案：** 在项目根目录新增 `_headers` 文件，CF Pages 会自动应用：

```
# _headers — Cloudflare Pages 静态资源缓存规则

/css/style.css
  Cache-Control: public, max-age=86400

/js/*
  Cache-Control: public, max-age=86400

/assets/*
  Cache-Control: public, max-age=604800

/index.html
  Cache-Control: public, max-age=0
```

保留现有的 `?v=X` 版本号机制作为缓存强制刷新手段。

**Railway 侧（API 响应头）：** 在 `backend/app.py` 中对各端点添加 `Cache-Control` 响应头：
- `/api/funds` → `max-age=60`（1 分钟，后端数据每 5 分钟刷新一次）
- `/health` → `max-age=30`
- `/api/funds/<code>/chart` → `max-age=300`（5 分钟）

**涉及文件：**
- `_headers`（新建，项目根目录）
- `backend/app.py`（各路由添加响应头）

**验收标准：**
- [ ] 浏览器 DevTools Network 面板确认静态资源返回 `Cache-Control` 头
- [ ] 重复访问时 JS/CSS 从缓存加载（200 from disk cache）
- [ ] API 响应有合理的缓存时间
- [ ] `index.html` 不会被过度缓存导致用户看不到更新
- [ ] 版本号 `?v=X` 升级后缓存失效正常

**预计耗时：** 1h
**依赖：** 无

## M4 代码重构

> 目标：降低维护成本、优化加载性能、为后续功能开发铺路。优先级低于功能开发。

### 4.1 JS 模块拆分

**当前问题：** `app.js` 约 1500 行，`LofFundMonitor` 类承担了数据获取、筛选排序、PC 表格渲染、移动端卡片渲染、排行榜、详情弹窗、图表、收益弹窗、设置管理、深色模式、欢迎弹窗等所有职责。

**方案：** 拆分为 5 个模块：

| 模块 | 文件 | 职责 |
|------|------|------|
| Monitor | `js/monitor.js` | 数据获取、状态管理、筛选排序、自动刷新 |
| Table | `js/table.js` | PC 端表格渲染 |
| Mobile | `js/mobile.js` | 移动端卡片渲染 |
| Detail | `js/detail.js` | 基金详情弹窗 + Chart.js 图表 + tooltip |
| UI | `js/ui.js` | 设置弹窗、toast、深色模式、欢迎弹窗、套利帮助弹窗 |

**约束：**
- 保留 `config.js` 和 `api.js` 不变
- 各模块通过 `LofFundMonitor` 主实例通信（事件或直接引用）
- HTML 中引入顺序：`config.js → api.js → monitor.js → table.js → mobile.js → detail.js → ui.js`

**验收标准：**
- [ ] 所有现有功能正常工作
- [ ] 各模块文件不超过 500 行
- [ ] 无循环依赖
- [ ] 移动端和 PC 端功能回归通过

**预计耗时：** 4-6h
**依赖：** 建议在 M3 功能开发稳定后进行

---

### 4.2 innerHTML 模板化

**当前问题：** HTML 字符串拼接分散在 `createFundRow()`、`createMobileCard()`、`_arbTooltip()`、`showProfitDetail()`、`showArbHelp()` 等方法中，维护和调试困难。

**方案：** 提取模板函数到独立文件或使用 template literals 集中管理。不引入额外库，使用原生 JavaScript。

**验收标准：**
- [ ] 所有 HTML 模板集中在独立的模板文件或模板对象中
- [ ] 修改 UI 结构时不需要搜索全部 JS 文件
- [ ] 功能回归通过

**预计耗时：** 2-3h
**依赖：** 4.1（先拆分再模板化更合理）

---

### 4.3 JS/CSS 压缩与构建

**当前问题：** JS 和 CSS 未压缩，总体积较大（app.js 76KB + style.css 47KB + config.js 1.5KB）。

**方案：**
- 引入轻量构建步骤（推荐 esbuild，零配置高速度）
- 输出文件：`dist/js/app.min.js`、`dist/css/style.min.css`
- 开发时仍使用原始文件，部署时使用压缩版本
- `package.json` 添加 `build` 脚本

**验收标准：**
- [ ] JS 总体积压缩到 < 40KB（压缩前 ~78KB）
- [ ] CSS 压缩到 < 30KB（压缩前 ~47KB）
- [ ] 压缩后功能与原始版本完全一致
- [ ] 构建命令 `npm run build` 一键执行

**预计耗时：** 1-2h
**依赖：** 4.1 + 4.2（拆分和模板化后再构建更合理）

---

### 4.4 Chart.js 按需加载

**当前问题：** Chart.js（~200KB）通过同步 `<script>` 标签加载，阻塞首页渲染。但图表仅在基金详情弹窗中使用，首屏不需要。

**方案：** 在 `showFundDetail()` 中动态加载 Chart.js，从 HTML 中移除同步 `<script>` 标签。

**涉及文件：** `index.html`、`js/detail.js`（拆分后的详情模块）

**验收标准：**
- [ ] 首页加载不请求 Chart.js
- [ ] 首次打开基金详情弹窗时动态加载 Chart.js
- [ ] 后续打开详情弹窗复用已加载的 Chart.js
- [ ] Chart.js 加载失败时图表区域显示降级提示

**预计耗时：** 30 min
**依赖：** 4.1（JS 拆分后 detail.js 独立）
